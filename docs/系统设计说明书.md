### 1.2 项目背景

#### 1.2.1 赛题背景

本届软件杯赛题聚焦"高等教育个性化学习资源体系"的建设。赛题核心洞察在于：**在数字化与智能化深度融合的时代，传统标准化教学模式难以适配学生在知识基础、认知风格、学习节奏上的显著差异**。赛题明确要求参赛团队构建多智能体系统，实现五大核心能力：

| 赛题要求 | 说明 |
|---------|------|
| **对话式学习画像构建** | 不少于 6 个维度，支持随学随新 |
| **多智能体协同资源生成** | 不同角色 Agent 协作产出至少 5 种多模态资源 |
| **个性化学习路径与推送** | 基于画像的路径规划与多类型资源精准推送 |
| **智能辅导（加分项）** | 多模态答疑，即时反馈与学习引导 |
| **学习效果评估（加分项）** | 多维度评估，动态调整学习方案 |

此外，赛题提出非功能性约束：界面美观流畅、流式输出、防幻觉机制、响应时间可控，以及**开发过程中使用的 AI 辅助工具需选用科大讯飞相关工具**。

#### 1.2.2 我们的回答

智学派系统以"**以学生画像为中心，多智能体协同驱动**"为设计理念，将五大核心能力落地为五个功能模块。系统以计算机专业核心课程《机器学习》为初始知识库，基于 Neo4j 知识图谱 + Qdrant 向量检索 + Chroma 对话记忆的多存储协同架构，在 GraphOrchestrator DAG 编排引擎的调度下，实现从"感知学生特征 → 生成个性化资源 → 规划学习路径 → 智能辅导答疑 → 评估效果反馈"的完整闭环。

**与赛题的核心呼应**：

- **画像 6+ 维度**：知识基础、认知风格、易错偏好、学习步调、内容偏好、目标导向，均含取值与置信度，并支持对话构建 + 答题反馈驱动的持续更新
- **7 种多模态资源**：课程文档、思维导图、练习题、教学 PPT、交互课件、代码案例、教学视频，由 7 个专职生成 Agent 并行产出
- **多智能体协同框架**：自研 DAG 编排引擎，RequireAgent → DesignAgent → 7×Generator → ReviewAgent，支持并行扇出、条件路由与最多 3 次质检重试
- **流式 + 防幻觉**：SSE 进度推送、L1 关键词过滤 + L2 大模型审核、来源标注与知识边界声明
- **讯飞生态集成**：系统架构预留讯飞星火大模型与 SeeDance 多模态生成接口，通过可插拔的 ModelRegistry 按配置启用

#### 1.2.3 技术驱动

系统以大模型技术为核心引擎，融合知识图谱、向量检索、多模态生成等前沿 AI 技术，构建"感知—生成—推送—评估—再感知"的闭环学习体系。技术选型遵循三项原则：优先选择社区活跃的开源框架、优先选择与 Java/Spring 生态无缝集成的组件、在满足需求的前提下控制部署复杂度。

### 1.4 术语定义

| 术语 | 英文 | 说明                                                                         |
|------|------|----------------------------------------------------------------------------|
| 智能体 (Agent) | Agent | 具备特定角色与能力的 AI 单元，接收输入状态、产出输出状态。本系统中每个 Agent 实现统一的 AgentNode 接口             |
| 多智能体系统 | Multi-Agent System | 多个专业 Agent 在 DAG 编排引擎下分工协作，通过共享不可变状态对象传递数据                                 |
| 学生画像 | Student Profile | 描述学生知识基础、认知风格、学习偏好等 6 个维度的结构化特征，每维含取值与置信度                                  |
| 知识图谱 | Knowledge Graph | 以 Neo4j 节点与关系组织的课程知识结构，表达知识点间的先修（REQUIRES）、关联（RELATED_TO）、包含（CONTAINS）等语义关系 |
| RAG | Retrieval-Augmented Generation | 检索增强生成：从外部知识库检索相关文档片段，注入大模型上下文后再生成，抑制幻觉、提升准确性                              |
| SSE | Server-Sent Events | 服务端单向推送事件流，浏览器原生支持，用于资源生成进度与推送通知的流式传输                                      |
| DAG | Directed Acyclic Graph | 有向无环图，本系统资源生成流程的编排模型：节点=Agent，边=执行顺序，扇出=并行，路由=条件分支                         |
| 记忆引擎 | Memory Engine | 基于 Chroma 向量数据库的对话记忆系统，存储原始对话并支持语义召回，配合滑动窗口策略管理上下文长度                       |
| 流式输出 | Streaming | 数据逐块生成并实时传输，前端边接收边渲染，降低首字节时延与用户感知等待                                        |
| 防幻觉 | Anti-Hallucination | 通过 L1 关键词审查 + L2 大模型审核 + 来源标注分级（私库/图谱/联网）+ 知识边界声明四层机制，抑制大模型在教育场景下生成事实性错误或违规内容 |

## 二、系统总体设计

### 2.1 功能模块划分

系统在功能上划分为五大核心模块，以"**以学生画像为中心**"的设计思想串联。画像模块产出六维学情特征，为资源生成、路径规划、辅导与推送提供个性化依据；各功能模块在使用过程中回流学习行为数据（答题结果、对话记录、资源访问），反哺画像持续更新，形成"感知→生成→推送→评估→再感知"的闭环。

```
                    ┌──────────────────┐
                    │   学习画像模块     │  ← 感知层
                    │  六维特征 + 置信度  │
                    └────────┬─────────┘
                             │ 画像驱动
          ┌──────────────────┼──────────────────┐
          ▼                  ▼                  ▼
   ┌──────────────┐  ┌──────────────┐  ┌──────────────┐
   │ 资源生成模块  │  │ 路径规划模块  │  │ 智能辅导模块  │  ← 执行层
   │ 7类型×7Agent │  │ BFS拓扑路径   │  │ 三级检索增强  │
   └──────┬───────┘  └──────┬───────┘  └──────┬───────┘
          │                  │                  │
          └──────────────────┼──────────────────┘
                             │ 行为数据回流
                             ▼
                    ┌──────────────────┐
                    │   学习评估模块     │  ← 反馈层
                    │ 掌握度+置信度追踪  │
                    └──────────────────┘
```

五大模块与赛题要求的对应关系：

| 模块 | 赛题要求 | 核心职责 | 关键类 |
|------|---------|---------|--------|
| 学习画像 | 功能1：对话式画像构建 | 六维特征提取、增量更新、置信度管理 | `ChatProfileService`, `ProfileExtractionAgent`, `ProfileUpdateTrigger` |
| 资源生成 | 功能2：多智能体资源生成 | DAG编排7类多模态资源并行生成与质检 | `ResourceGenerationService`, `GraphOrchestrator`, `*Generator` |
| 路径规划 | 功能3：学习路径与推送 | 基于KG拓扑的个性化路径规划与资源推荐 | `LearningPathService`, `ResourceRecommendService` |
| 智能辅导 | 功能4：智能辅导（加分项） | 三级检索增强问答、滑动窗口记忆 | `AiTutorServiceImpl`, `ConversationMemoryService` |
| 学习评估 | 功能5：效果评估（加分项） | 掌握度追踪、画像触发更新、学情概览 | `ProficiencyService`, `ProfileUpdateTrigger` |

### 2.2 技术架构分层


```
┌─────────────────────────────────────────────┐
│  表现层 (Presentation)                       │
│  Vue 3 SPA · Element Plus · SSE EventSource  │
│  marked.js/KaTeX/Mermaid 多模态渲染          │
├─────────────────────────────────────────────┤
│  应用服务层 (Application Service)            │
│  REST Controllers · SSE Stream API           │
│  请求校验 · Token鉴权 · 全局异常处理          │
├─────────────────────────────────────────────┤
│  智能体与领域服务层 (Agent & Domain)          │
│  ┌──────────┬──────────┬──────────┐         │
│  │ DAG编排  │ 画像Agent│ 辅导Agent│         │
│  │ 7×生成器 │ 路径规划 │ 评估服务 │         │
│  └──────────┴──────────┴──────────┘         │
│  ModelRegistry · Provider抽象 · 记忆引擎     │
├─────────────────────────────────────────────┤
│  数据与外部服务层 (Data & External)           │
│  MySQL · Neo4j · Qdrant · Chroma · Redis     │
│  DeepSeek/讯飞 API · Seedance · DuckDuckGo   │
└─────────────────────────────────────────────┘
```

**核心设计原则**：

| 原则 | 说明 | 实现方式 |
|------|------|---------|
| **响应式非阻塞** | 长耗时 LLM 调用与向量检索不阻塞事件循环 | WebFlux/Netty + boundedElastic 线程池调度 |
| **多模型可插拔** | 文本/视觉/视频任务可路由到不同厂商模型 | `ModelRegistry` 统一抽象，按 `application.yml` 配置切换 |
| **存储职责分离** | 不同类型数据使用最适合的存储引擎 | 关系型→MySQL，图→Neo4j，文档向量→Qdrant，对话记忆→Chroma，缓存→Redis |
| **不可变状态传递** | Agent 间通过不可变 record 传递上下文，杜绝并发写入 | `GenerationState` (Java record) + `withX()` 拷贝方法 |
| **局部降级兜底** | 任一外部依赖故障不影响主流程 | try/catch 返回空结果 + Fallback 终态 |

**为什么选择四层架构而非简单的 MVC？** 赛题要求的多智能体协同本质上是一个复杂的业务流程编排问题。将 Agent 编排与领域逻辑独立为中间层，使得：
1. 应用服务层专注于 HTTP 协议适配（REST/SSE），不被业务逻辑污染；
2. Agent 层可以独立测试、独立扩展（新增资源类型只需新增 Generator），不牵动上层；
3. 数据层的外部依赖替换（如从 Chroma 切换到 Milvus）对其他层透明。

### 2.3 技术选型说明

选型总体遵循三条原则：**① 优先选择社区活跃、生态成熟的开源框架；② 优先选择与 Java/Spring 生态无缝集成的组件；③ 在满足功能与性能要求的前提下控制资源占用与部署复杂度。**

#### 2.3.1 前端技术选型

| 技术 | 选型 | 选型理由 |
|------|------|---------|
| 框架 | **Vue 3 (Composition API)** | 响应式数据流清晰、`<script setup>` 语法简洁、单文件组件开发效率高；相比 React 学习曲线更平缓，适合团队快速上手 |
| 构建工具 | **Vite 8** | 基于 ESBuild 的秒级冷启动与 HMR 热更新，开发体验显著优于 Webpack；原生 ESM 支持 |
| UI 组件库 | **Element Plus 2** | 组件丰富（70+）、中文文档完善、与 Vue 3 深度适配；选择而非 Naive UI 是因为 Element Plus 社区更大，遇到问题更容易找到解决方案 |
| 路由 | **Vue Router 4** | Vue 官方路由方案，支持 history 模式与全局前置守卫（`beforeEach`）实现 Token 鉴权 |
| HTTP | **Axios** | 拦截器机制便于统一注入 Token 与响应错误处理；比 fetch 更丰富的超时/取消/重试配置 |
| 渲染增强 | **marked + KaTeX + Mermaid** | 分别负责 Markdown 富文本、LaTeX 数学公式、流程图/思维导图的前端渲染；三者均为纯前端方案，无需服务端渲染 |
| 实时通信 | **EventSource (SSE)** | 轻量、浏览器原生支持、自动重连；符合资源生成进度与推送通知的单向流式推送场景，无需 WebSocket 的双向复杂度 |

#### 2.3.2 后端技术选型

| 技术 | 选型 | 选型理由 |
|------|------|---------|
| 语言/运行时 | **Java 21** | LTS 长期支持，虚拟线程（Virtual Threads）与 record 类型显著提升并发处理与数据建模效率 |
| 应用框架 | **Spring Boot 3.4.3** | 生态成熟、自动装配、与各数据源（JPA/Neo4j/Redis）集成便捷；国内企业级开发事实标准 |
| Web 框架 | **Spring WebFlux (Netty)** | 响应式非阻塞模型，天然适配 SSE 流式输出与高并发长连接场景 |
| AI 编排 | **Spring AI 1.0.0-M6 + LangChain4j 1.6.0** | 统一大模型调用抽象（`ChatClient`）与 Agent 编排能力（`@AiService`），减少样板代码 |
| 构建工具 | **Maven** | 依赖管理规范、插件生态完善；团队对 Maven 更熟悉 |
| 鉴权 | **Token + BCrypt** | BCrypt 加盐哈希存储密码，Token 承载无状态登录态 |

**为什么选择 WebFlux 而非 Spring MVC？** 系统存在大量长耗时的大模型调用（单次 10–20 秒）与流式输出场景。传统 Servlet 模型每个请求独占一个线程，高并发时线程池迅速耗尽；WebFlux 的响应式模型基于 Netty 事件循环，少量线程即可支撑大量并发连接。AI 调用等阻塞操作被显式调度至 `boundedElastic` 线程池隔离，避免阻塞事件循环。

#### 2.3.3 大模型与多模态技术选型

| 能力 | 选型 | 选型理由 |
|------|------|---------|
| 文本大模型 | **DeepSeek (deepseek-chat)** | 中文理解与推理能力强、API 兼容 OpenAI 规范、成本可控（约为 GPT-4o 的 1/10）；作为默认文本 Provider |
| 文本嵌入 | **智谱 Embedding-3 (2048维)** | 中文语义表征优秀（C-MTEB 榜单领先），2048 维在精度与存储开销间平衡；单批最多 4 条，适合对话级粒度 |
| 视频生成 | **Seedance (doubao-seedance-1-5-pro)** | 文生视频质量高，配合 Seedream 图像模型生成教学分镜；由科大讯飞生态提供 |
| 语音合成 | **阿里云 NLS TTS** | 中文发音自然，支持多种音色；预留集成接口 |
| 多模型路由 | **自研 ModelRegistry** | 统一 `ModelProvider` 抽象，支持按任务类型（text/visual/video）路由到不同厂商，通过配置开关避免厂商锁定 |

**为什么自研 ModelRegistry 而不是直接用 Spring AI 的自动配置？** Spring AI 的自动配置假定单一 Provider，而本系统需要在文本生成、图像生成和视频生成三个场景使用不同的模型厂商。自研路由层将 Provider 选择从业务逻辑中解耦，增加新厂商（如讯飞星火）只需配置 `application.yml` 和新增一个 Provider 实现类，无需修改业务代码。

#### 2.3.4 数据存储技术选型

| 存储 | 选型 | 用途 | 选型理由 |
|------|------|------|---------|
| 关系数据库 | **MySQL 8** | 用户账户、六维画像、知识点掌握度、画像会话 | 结构化数据、ACID 事务、JPA 成熟支持 |
| 图数据库 | **Neo4j** | 机器学习课程知识图谱 | 先修/关联/包含等关系天然是图结构；Cypher 查询比 SQL 递归 CTE 更简洁高效 |
| 文档向量库 | **Qdrant** | 私人智库文档向量检索 | 支持按 `userId` payload 过滤的用户隔离、余弦距离、gRPC 高性能 |
| 对话记忆库 | **Chroma** | AI 辅导对话历史向量存储 | 轻量嵌入式部署、REST API 简单、适合按会话组织的时序对话数据 |
| 缓存 | **Redis** | 资源生成检查点（断点续跑） | 高性能、TTL 自动过期、适合临时状态存储 |

**为什么用两套向量数据库（Qdrant + Chroma）而非统一？** 两种数据的访问模式不同：文档检索需要按 `userId` 过滤 + 高吞吐 QPS + gRPC 性能，Qdrant 的 payload 过滤与 gRPC 接口更优；对话记忆需要按 `conversationId` 组织、按时间排序、轻量部署，Chroma 的 REST API 更简单且资源占用更低。职责分离也避免单点故障影响两个模块。

#### 2.3.5 部署与运维技术选型

| 维度 | 选型 | 说明 |
|------|------|------|
| 容器化 | **Docker + Docker Compose** | 前后端与各数据库打包为独立容器，一键编排启动；docker-compose.yml 定义服务/网络/数据卷 |
| 前端部署 | **Nginx 静态托管 + 反向代理** | 托管 Vue 打包产物并将 `/api` 反向代理至后端 8080 端口 |
| 配置管理 | **application.yml + .env** | 敏感密钥经 dotenv-java 注入，.env 不入库不入镜像 |
| 定时任务 | **Spring @Scheduled** | 知识图谱每周日构建 |

### 2.4 Agent 职责总览

系统以多智能体协同为核心组织方式。各 Agent 职责单一、边界清晰，通过 DAG 编排引擎统一调度（资源生成流程）或方法调用直接协作（画像/辅导/评估流程），通过不可变状态对象传递数据。

```
资源生成流程（DAG编排）：
  RequireAgent → DesignAgent → ┬ DocGenerator
                               ├ MindmapGenerator
                               ├ QuizGenerator
                               ├ CodeGenerator    ← 并行扇出
                               ├ HtmlGenerator
                               ├ PptGenerator
                               └ VideoGenerator
                                     ↓
                               ReviewAgent → DONE / 重试→DESIGN

画像流程（事件驱动）：
  ChatProfileService → ProfileExtractionAgent → ProfileUpdateAgent
  QuizSubmitController → ProficiencyService → ProfileUpdateTrigger → ProfileAnalysisService

辅导流程（直接调用）：
  AiTutorServiceImpl → ConversationMemoryService
                     → RagService → KgService → WebSearchService
```

| Agent | 所属模块 | 核心职责 | 关键实现类 |
|------|---------|---------|-----------|
| 画像提取 Agent | 学习画像 | 从对话/表单中提取六维画像特征，输出结构化 JSON（取值+置信度） | `ProfileExtractionAgent` |
| 画像分析 Agent | 学习画像 | 从 Chroma 对话记录批量分析并更新画像 | `ProfileUpdateAgent` (LangChain4j) |
| 画像更新触发器 | 学习画像 | 追踪对话轮数，达到阈值（7轮或3天）时触发画像重分析 | `ProfileUpdateTrigger` |
| 需求分析 Agent | 资源生成 | 汇聚画像、知识图谱、RAG 上下文形成生成需求 | `RequireAgent` |
| 设计 Agent | 资源生成 | 调用 LLM 生成资源蓝图（结构大纲、各类型内容要点、个性化调整） | `DesignAgent` |
| 文档/思维导图/题库/代码/HTML/PPT/视频生成 Agent | 资源生成 | 按蓝图并行生成七类多模态资源 | `DocGenerator`, `MindmapGenerator`, `QuizGenerator`, `CodeGenerator`, `HtmlGenerator`, `PptGenerator`, `VideoGenerator` |
| 质检 Agent | 资源生成 | L1 关键词过滤 + L2 LLM 准确性审核，不通过回退重生成（最多3次） | `ReviewAgent` |
| DAG 编排引擎 | 资源生成 | 解析 GenerationGraph，顺序执行节点、并行扇出、条件路由、Redis 检查点 | `GraphOrchestrator` |
| 路径规划 Agent | 学习路径 | 基于 Neo4j 知识图谱 BFS 拓扑排序生成个性化学习路径 | `LearningPathService` |
| 资源推荐 Agent | 学习路径 | 综合画像偏好、薄弱点、学习进度三维生成每日推荐 | `ResourceRecommendService` |
| 辅导 Agent | 智能辅导 | 三级检索增强（RAG→KG→Web）的问答生成 | `AiTutorServiceImpl` |
| 记忆引擎 | 智能辅导 | 滑动窗口分层记忆：近期完整原文 + 远期 LLM 摘要 | `ConversationMemoryService` |
| 知识点评估 Agent | 学习评估 | 掌握度计算（correct/total）+ 置信度计算（1-e^(-kn)） | `ProficiencyService` |

**为什么没有设计一个统一的"调度中枢 Agent"？** 资源生成、智能辅导、画像更新三类任务的编排需求差异显著：资源生成需要 DAG 并行扇出 + 重试回退，辅导需要三级检索链的优先级切换，画像更新需要轮数/时间双条件触发。强行用一个统一调度器适配三种场景会导致调度器本身过度复杂。本系统采用"领域内自治编排"策略——每种任务类型选择最适合的编排模型（DAG/责任链/事件驱动），Agent 之间通过方法调用直接协作，更符合单一职责原则。

**Agent 间通信方式**：同一资源生成流程内的节点通过共享不可变 `GenerationState` record 传递上下文；跨模块 Agent（如辅导→画像触发）通过 Spring 依赖注入直接方法调用；对外统一以 SSE 向前端推送进度事件。
